That's a good question. The weights I suggested are not a standard—they were an example based on the relative importance of each metric in measuring security behavior. The weights should reflect how much each behavior contributes to actual cybersecurity risk.

Here's the reasoning behind each one:

Metric	Why it Matters	Suggested Weight
Training Completion	Training improves awareness, but completing training doesn't guarantee secure behavior.	20%
Email Open Rate	Simply opening an email isn't necessarily risky because legitimate emails are opened too.	10%
Safe Link Behavior	Clicking a phishing link is a direct indicator of susceptibility to phishing.	25%
Credential Safety	Submitting credentials to a phishing page is the most severe failure because it can directly compromise accounts.	35%
Repeat Offender Penalty	Repeated failures indicate persistent risky behavior and should affect the score.	10%
Why is Credential Safety weighted highest?
Consider the impact of different actions:

Opened phishing email → No real compromise

Clicked phishing link → Potential exposure

Submitted credentials → Actual credential compromise

The severity increases at each step, so credential submission deserves the highest weight.

For example:

Employee	Opened	Clicked	Submitted Credentials	Risk
A	✅	❌	❌	Low
B	✅	✅	❌	Medium
C	✅	✅	✅	High
Employee C represents a much greater security risk than Employee B, even though both clicked the link.

A Better Approach: Base the Weights on Risk Severity
Instead of choosing weights arbitrarily, assign points based on the severity of each action.

Event	Risk Points
Completed training	+20
Ignored phishing email	+30
Reported phishing email	+20
Clicked phishing link	-20
Submitted credentials	-40
Repeat offense	-10 per occurrence
Then calculate:

Department Security Score =
100
+ Positive Points
- Negative Points
Finally, clamp the score between 0 and 100.

This method is easier to explain because each event has a clear impact on the score.

An Even More Data-Driven Approach (Recommended for a Final-Year Project)
Rather than assigning fixed weights, derive them from your organization's priorities. For example:

Objective	Weight

Reduce repeat offenders	10%
The department score then becomes:

Security Score =
0.40 × Credential Safety
+ 0.30 × Safe Link Behavior

+ 0.10 × Repeat Offender Score
This approach is easier to justify because the weights are directly tied to business objectives.

Which approach should you use?
For an academic project, I recommend the objective-based weighted model. It is:

In Java:

for (SuspiciousWord word : suspiciousWords) {
    if (emailContent.toLowerCase().contains(word.getWord().toLowerCase())) {
        matchedWords.add(word);
        score += word.getPoints();
    }
}
This method is easy to implement using Spring Boot + MySQL.

Option 2: Regular Expressions (Regex)
Some phishing indicators have many variations.

Instead of searching only for:

password
you could search for:

pass(word)?
or

verify|verification|verified
Example:

Pattern pattern = Pattern.compile("verify|verification", Pattern.CASE_INSENSITIVE);
Matcher matcher = pattern.matcher(email);

if (matcher.find()) {
    // matched
}
Regex is useful when you want to detect multiple forms of the same word.

This is a more advanced approach.

Instead of matching exact words, NLP tries to understand the text.

Example:

Apache OpenNLP

3. How redirects actually work (core idea)

Step B: Backend controls redirects
In Spring Boot:

@GetMapping("/track/{campaignId}/{userId}")
public void track(@PathVariable Long campaignId,
                  @PathVariable Long userId,
                  HttpServletResponse response) throws IOException {

    int redirectCount = campaignService.getRedirectCount(campaignId);

    // Log click
    trackingService.saveClick(campaignId, userId);

    // Redirect loop simulation
    response.sendRedirect("/redirect/step1?c=" + campaignId + "&u=" + userId);
}
Step C: Multiple redirect steps
@GetMapping("/redirect/step{n}")
public void redirect(@RequestParam int c,
                     @RequestParam int u,
                     HttpServletResponse response) throws IOException {

    int totalRedirects = campaignService.getRedirectCount(c);

    int currentStep = trackingService.getCurrentStep(u, c);

    if (currentStep < totalRedirects) {
        trackingService.incrementStep(u, c);

        response.sendRedirect("/redirect/step" + (currentStep + 1)
                + "?c=" + c + "&u=" + u);
    } else {
        response.sendRedirect("/landing-page?c=" + c + "&u=" + u);
    }
}
